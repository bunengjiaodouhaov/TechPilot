# TechPilot 学习与协作协议

## 1. 总目标
- 面试导向
- 结果导向
- 项目必须可解释、可验证、可复现
- 不以机械复制代码作为完成标准

## 2. 每个模块的固定顺序

1. **产品视角**：它是什么，用户为什么需要？
2. **数据视角**：需要保存什么，实体关系是什么？
3. **系统视角**：它位于架构中的哪里，与哪些模块协作？
4. **实现视角**：阅读、编写或修改代码。
5. **验证视角**：运行、测试并保留完成证据。
6. **面试视角**：解释设计理由、替代方案、取舍和失败处理。

如果用户表示不知道某个概念，应暂停实现，先解释：
- 它是什么
- 为什么存在
- 它解决什么问题
- 它在系统中的位置

## 3. 代码能力目标
当前阶段不要求脱离文档默写框架 API，但必须逐步达到：
- 能读懂代码
- 能解释每个模块和关键语句的职责
- 能发现明显错误
- 能修改小范围逻辑
- 能把业务需求拆成函数和步骤

训练节奏：
- 前期：设计、阅读、修改
- 中期：补全小函数
- 后期：独立完成局部模块

## 4. 回答与纠错规则
- 不会就明确说不会。
- 可以猜，猜错后通过推理修正。
- 发现错误时直接指出结论，不使用模糊措辞掩盖问题。
- 区分“必须理解”和“知道即可”的内容。
- 不用空泛鼓励替代技术反馈。

## 5. 项目状态来源
默认按以下优先级恢复上下文：

1. 总控手册：确定当前阶段和验收要求
2. GitHub `main`：确认已提交、已推送的真实代码状态
3. `docs/PROJECT_STATUS.md`：确认当前进度
4. `docs/DEV_LOG.md`：了解历史决策、错误和修复
5. 用户明确说明的本地未推送修改

除非用户说明本地存在未 push 的修改，否则不要求用户重复粘贴 GitHub 已有代码。

## 6. 每日收尾规则
当日验收项完成后，自动执行以下收尾流程，不等待用户额外要求：

1. 对照总控手册检查遗漏
2. 运行测试与关键验证命令
3. 更新 `PROJECT_STATUS.md`
4. 更新 `DEV_LOG.md`
5. 必要时更新 `RUNBOOK.md`
6. 更新 `INTERVIEW_NOTES.md`
7. 自动生成当日 Review
8. 检查 `git status` 和 `git diff`
9. Commit 并 Push
10. 明确宣布当日结论：PASS / CONDITIONAL PASS / FIX

## 7. 当日 Review 规则
- Review 只保留本地，不提交 GitHub。
- 建议路径：`.local/reviews/dayXX-review.md`
- `.local/` 必须加入 `.gitignore`
- Review 至少包含：
  - 今日完成
  - 系统链路
  - 必须理解
  - 不要求背诵
  - 错误与修复
  - 面试问答
  - 明日第一任务

## 8. 防跑偏
- 严格按总控手册当前阶段推进。
- 不擅自改变 Day 的目标或拆出额外功能。
- 需要补充原理时，只补充当前步骤相关内容。
- 代码完成不等于 Day 完成；文档、测试、GitHub 管理和验收必须一致。
